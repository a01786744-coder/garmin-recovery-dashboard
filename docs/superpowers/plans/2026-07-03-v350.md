# v3.5.0 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans (inline).
> Spec = the user's approved prompt (2026-07-03). Steps use checkbox syntax.

**Goal:** Reliability fixes, tray/background mode, score transparency + strain
history, journal depth — one release (v3.5.0), screenshots shown to the user
BEFORE tagging.

**Global constraints:** local-first; no fabricated data; never crash on Garmin
errors; credentials never logged; TDD for backend logic; verify against the
real app; existing 144 backend tests stay green. Release only after the user

---

## Phase 1 — Reliability

- [ ] **1.1 Day-list refresh** (already edited, uncommitted): `App.jsx` `load()`
  fetches `getDays()` every cycle; drop the now-duplicate one-shot fetch in the
  auth effect. Build, commit.
- [ ] **1.2 Stale-data banner**: in `App.jsx`, when the newest data date is >2
  days old, show a dismissible amber banner "No new data since <date> — check
  that your watch is syncing." Dismiss stores the date in localStorage so it
  reappears only if staleness advances. Frontend-only.
- [ ] **1.3 Auth self-healing**: TDD — `db.last_sync_statuses(path, n)` returns
  the last n `(status, message)`; `/api/auth/status` adds
  `needs_relogin: true` when the 3 most recent syncs are all
  `error/GarminAuthError`. Frontend: when `needs_relogin`, show the Login
  screen with a notice "Session expired — sign in again" (new `notice` prop).

## Phase 2 — Background/tray mode

- [ ] **2.1 Backend setting** (TDD): `start_at_login` (bool, default false) in
  `settings.py`; checkbox in `Settings.jsx` ("Start at login (minimized)").
- [ ] **2.2 Electron tray + close-to-tray** (`electron/main.js`):
  - `Tray` with the app icon (packaged: `resourcesPath/frontend/icon-192.png`;
    dev: `frontend/public/icon-192.png`), tooltip, menu: Open Dashboard / Quit.
  - Window `close` hides instead of quitting (backend stays up); `Quit` sets
    `app.isQuitting` and really exits (killBackend as today).
  - `requestSingleInstanceLock`: second launch shows the existing window.
  - Login-item: read `settings.json` from userData at startup + `fs.watch` it;
    apply `app.setLoginItemSettings({openAtLogin, openAsHidden:true,
    args:["--hidden"]})`. When launched with `--hidden` (or
    `wasOpenedAsHidden`), create the window hidden.
  - macOS: window close hides; dock stays; Tray = menu-bar icon.
  Manual verification on Windows (close → tray alive → phone still served →
  reopen from tray → Quit stops backend).

## Phase 3 — Transparency & strain history

- [ ] **3.1 Recovery explanation** (TDD): `recovery.recovery_explanation(hrv,
  rhr, hrv_hist, rhr_hist, min_days)` → `{hrv: {today, baseline, z}, rhr:
  {today, baseline, z}, weights: {hrv: .7, rhr: .3}}` or None (same guards as
  the score). `/api/today` adds `recovery_explain`. Frontend: in the Recovery
  detail panel, an "Why this score" block: two rows (HRV/RHR) with
  today-vs-baseline and an up/down chip; weights caption.
- [ ] **3.2 Strain breakdown + tuning** (TDD): `recovery.strain_breakdown(acts,
  m)` → `{workout, daily, total, score}`; `strain_score` delegates to it.
  Tuning: when workout load > 0, daily-life load counts at **50%** (workouts
  already contain their own steps/intensity — reduces double counting).
  Target tests: rest day (2k steps) ≤ 8; walk day (12k steps + 30 mod min)
  25–35; run day (load 90 + 12k steps + 40 vig min) 55–70. `/api/today` adds
  `strain_explain`; Strain detail panel shows workout vs daily split.
- [ ] **3.3 Strain history backfill** (TDD): `fetch_baseline` also parses
  steps, intensity_moderate, intensity_vigorous, active_calories, distance_m
  from the user summary it ALREADY fetches (no new API calls). Bump
  `BASELINE_FETCH_VERSION` → 3 (one-time paced re-backfill). Rescore then
  fills historical strain; heatmap/trend fill.

## Phase 4 — Journal & insights depth

- [ ] **4.1 Past-day journal**: `Overview.jsx` renders `<Journal date={m.date}>`
  for the currently viewed day (live or browsed) — API is already
  date-generic. Journal card title shows the date when it isn't today.
- [ ] **4.2 Sleep correlations + chip badges** (TDD): `journal_correlations`
  also tests each tag against **next-night sleep_score** (sleep is filed under
  the wake date = entry date + 1); items gain structured `tag` + `metric`
  fields. `Journal.jsx` receives `insights` and marks tag chips that have a
  known effect with ▲/▼.
- [ ] **4.3 Monday weekly recap** (TDD): `insights.week_extremes(daily)` →
  best/worst recovery day of the last 7 (None-safe). Today tab shows a
  "Your week in review" card on Mondays: weekly averages vs last week
  (existing `weekly`), best/worst day, count of active correlations.
